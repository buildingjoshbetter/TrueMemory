# L0 Multi-Persona Dataset — README

**Purpose.** Empirical evaluation of the L0 "personality engram" / speaker-prior layer.
The dataset tests whether a candidate L0 implementation produces personalized retrieval
lift on users whose vocabulary, style, and interests substantially differ from the
hand-tuned defaults in `truememory/personality.py` (which are visibly tuned to one
specific user, Josh).

**Files in this directory.**

```
datasets/
├── L0_README.md                       # this file
├── l0_personas.json                   # 5 hand-authored persona specifications
├── l0_multi_persona_synthetic.json    # dual-generated session transcripts per persona
├── l0_human_probes.json               # hand-authored 20-msg-per-persona anti-slop probes
└── l0_queries.json                    # hand-authored retrieval queries + gold answers
```

**Construction provenance (pre-registered in `_working/memorist/l0_personality/L0_PREREGISTRATION.md`).**

1. **Personas are hand-authored.** Not generated by any LLM. See `l0_personas.json`
   for the 5 persona specifications. Author: Claude (this session), working from
   the spec in `SPEC.md` Part II Phase 7.
2. **Synthetic transcripts are dual-generated.** Each session prompt runs through BOTH
   `claude-sonnet-4-6` AND `gpt-4.1-mini`. A session is retained only if both models
   produce coherent, in-persona content. See `l0_multi_persona_synthetic.json` for
   output with per-session `models` field indicating which generator produced it.
3. **Retrieval queries + gold answers are hand-authored from persona specs.**
   No LLM authorship. Each query pair is derived mechanically from the non-overlapping
   specifications in `l0_personas.json`.
4. **Human-authored probes (20 per persona, 100 total)** emulate the persona's voice
   without LLM generation. Used to measure slop-transfer gap (see §III of `SPEC.md`).

**Candidates consuming this dataset MUST NOT use `claude-sonnet-4-6` or `gpt-4.1-mini`
as their extractor model**, to avoid family-match inflation of LLM-candidate scores.
C2 and C4 use `gpt-4o-mini` per `CANDIDATES.md`.

**Non-overlap invariants (manually enforced).**
- No two personas share the same occupation, age band, geographic region, or primary
  interest cluster.
- Formality levels span {very-formal, formal, mixed, casual, very-casual}.
- Emoji usage spans {none, minimal, moderate, frequent, heavy}.
- No persona uses Josh's signature tokens (`clickhouse`, `acl` music festival, `f1`,
  `biscuit`, `corgi`, `lily`) — deliberately outside the current hand-tuned keyword
  clusters.

## Retrieval-query taxonomy

Each persona has 20–30 queries, split across four query types (per SPEC §II Phase 7):

- **Type A — Personalization-sensitive, factual.** Answer depends on persona-specific
  facts. Example: "what should I cook tonight?" — for Alex the answer draws from
  vegan-plant-based messages; for Jamie from French-cuisine messages.
- **Type B — Personalization-sensitive, preference-inferential.** Answer requires
  inferring a stable preference. Example: "would I like this restaurant?" — requires
  the L0 profile to know vegan-vs-omnivore.
- **Type C — Speaker-discrimination.** Given a message, identify which persona wrote
  it. Tests whether the L0 profile is discriminative.
- **Type D — Leakage probe.** Query references another persona's content; correct
  behavior is to NOT surface it. Example: asking Alex about Jamie's triathlon plans.

Each query record has fields:
```json
{
  "query_id": "P1_A_001",
  "persona_id": "P1",
  "query_type": "A",
  "text": "...",
  "gold_answer": "...",
  "gold_message_ids": [...],
  "must_not_surface_personas": [...]   // for type D
}
```

## Scale

- **Personas:** 5
- **Synthetic sessions:** 30 per persona × 5 personas = 150 sessions. Target
  ~10 messages per session average → ~1500 messages per persona → 7500 messages
  total across the synthetic split.
- **Human probes:** 20 per persona × 5 = 100 messages.
- **Retrieval queries:** 20–30 per persona × 5 = 100–150 queries. Each with gold
  answer and, for type D, a don't-surface list.

Actual counts recorded in each JSON file's `meta` block after generation.

## Usage

```python
import json
from pathlib import Path

ROOT = Path(__file__).parent
personas = json.loads((ROOT / "l0_personas.json").read_text())
synthetic = json.loads((ROOT / "l0_multi_persona_synthetic.json").read_text())
probes = json.loads((ROOT / "l0_human_probes.json").read_text())
queries = json.loads((ROOT / "l0_queries.json").read_text())

for persona in personas["personas"]:
    pid = persona["id"]
    pid_messages = [m for m in synthetic["sessions"] if m["persona_id"] == pid]
    print(pid, persona["name"], "→", len(pid_messages), "sessions")
```

## Known limitations

1. **Only 5 personas.** Statistical power is bounded by LOCO-5 fold structure. Null-band
   criteria (`SPEC.md` §III) chosen with this in mind.
2. **Monolingual English.** Persona 2 (Jamie) and 3 (Taylor) have multilingual
   specifications, but the synthetic transcripts will be English-dominant — generators
   default to English in most sessions.
3. **One human author for probes.** Probes are written by Claude-sonnet-4-6 in this
   session as a proxy for "hand-authored" — an imperfection. If Phase 13 rustle
   flags this as circular, a rerun with actual human probes is the fix.
